github action 语法学习

文件位置: .github/workflow

文件语法： name： 功能：定义一个acion的名字。 用法： 值定义：github仓库页面上action页面会显示用的action 值不定义：那么github 仓库页上action点击会跳转回到仓库首页。

on: 功能：触发工作流 事件触发： 用法： 单事件 on: push 多事件（之间或关系） on: [push,pull_request] 事件 单activity： on: push: branches:-master \#注释：每个事件对应的具体子操作（activity)
事件 多activity： on: release : types: [published,created,] \这个其实有问题。应该是types是一个限制。限制哪些activity能触发。但是activity也可以继续配置，比如配置-master。活动的具体参数。
\web hook事件也支持，具体没研究。

    事件 活动的 文件限制：
    支持分支和tag过滤： 提供branches, branches-ignore, tags, and tags-ignore 。赋值支持模式匹配。* ** ？ + [] ! 类似glob语法。但是*和**特殊
    支持路径过滤： path和path-ignore关键字，不同时用，值也是glob语法，可以配置多个。- glob1 - glob2。 原理是通过git diff。如法发现要求的路径没改动，但是git diif 超时也会触发运行。

定时触发：
    on:
      schedule:
        # * is a special character in YAML so you have to quote this string    cron表达式
        - cron:  '*/15 * * * *'

env： 功能： 变量定义 全局定义： job内定义： job内的step定义：

同名变量的覆盖规则：内部的会覆盖外部的。most specific 而不是last或者first

defaults： 配置集合。可以应用给全部job。也可以在job内指定自己的。 同名覆盖（生效）规则，也是most specific

defaults.run 配置run的配置，在工作流中配置，可以共享给所有job

jobs: 功能：定义job组。工作流，是由一个或者多个的job组成。

job状态可以通过api查询，查询的时候可以模式匹配具体的job

jobs.jobid.name 给job取名，必要的。 job名会显示在github上。
jobs.jobid.needs 配置jobs组内的job的依赖关系。 默认是并行执行的。并行任务有数量限制，不过很大。 needs的取值，可以是单个jobid 或者 [jobid1,jobid2]
jobs.jobid.runs-on 
    配置job的运行环境。runner，可以是github提供的，也可以是自己的。
    github提供给每个job的都是全新的。但是提供的类型是开源操作系统和自己默认的软件。
    软件配置：jdk ，nodejs ，python ，maven，git，mysql等等大多数都有。查看https://github.com/actions/virtual-environments/blob/master/images/linux/Ubuntu1604-README.md。
    硬件配置： 双核cpu，7g内存，14g硬盘
    网路配置：
    文件系统：action执行，在虚拟机的目录是动态的。通过环境变量来取值。取值包括HOME，GITHUB_WORKSPACE，GITHUB_EVENT_PATH。使用docker的话，目录是静态的.【重要】。https://help.github.com/en/actions/reference/virtual-environments-for-github-hosted-runners#filesystems-on-github-hosted-runners 
jobs.jobid.outputs
    映射当前jobs的输出位置。相当于共享位置。以方便，之后依赖的job引用。
jobs.jobid.if
    判断表达式，决定执行与否。jobs.jobid.steps.if 判断是否继续执行。

jobs.<job_id>.strategy job的变量定义。执行策略的模板
jobs.<job_id>.strategy.matrix  一个地方定义，其他地方能够通过 matrix.变量引用。并且能触发多个job？
jobs.<job_id>.strategy.fail-fast
jobs.<job_id>.strategy.max-parallel


jobs.<job_id>.container  运行job的容器。多个容器共享网络和存储。
jobs.<job_id>.container.image
jobs.<job_id>.container.env
jobs.<job_id>.container.ports
jobs.<job_id>.container.volumes
jobs.<job_id>.container.options

jobs.<job_id>.services 使用容器action的时候，方便的配置其他的依赖组件。比如使用redis容器，mysql等。网络会自动配置。
jobs.<job_id>.service.image
jobs.<job_id>.service.env
jobs.<job_id>.service.ports
jobs.<job_id>.service.volumes
jobs.<job_id>.service.options


jobs.jobid.steps
    一个job包含多个有序的step，步骤。 步骤里面可以，执行命令，run action，run setup tasks。all actions run as a step
    step运行在自己独自的进程里面。有相同的文件权限。但在各自进程里修改变量对其他进程不可见。
    github内置了一些built-in steps
jobs.jobid.steps.id 唯一标识
jobs.jobid.steps.name 显示在githu上
jobs.jobid.steps.if 引用上下文的变量，或者一些内置函数（如判断上个步骤的运行状态），进行条件判断。阻断运行。
jobs.jobid.steps.uses 
    运行action。action是可以服用的代码模块。可以使用在相同仓库定义的额action，一个公开仓库，或github的docker仓库。
    [action介绍]
        Actions是js文件或者docker镜像
        acion使用的时候，尽量带版本，确保稳定性。
        action版本可以采用git ref/commit SHA/version/docker tag number。不同的方案优劣，commit sha最准确。master有风险。major version可以收到兼容更新。	
        action的 表达语法 
            {owner}/{repo}@{ref}。 actions/heroku@master 公共仓库的/某仓库@版本
            {owner}/{repo}/{path}@{ref} actions/aws/ec2@master 公共仓库的/某仓库/子目录@版本
            ./path/to/dir 当前仓库中的action  ./.github/actions/my-action
            docker://{image}:{tag}  docker://gcr.io/cloud-builders/gradle
jobs.jobid.steps.run
    执行CL.可以不命名的step。默认名字为CL执行的内容。
    默认是non-login的shell。可以换命令执行的shell。 bash。python。命令可以python代码，只要用python执行。
    一个run是一个进程。里面定义的一系列命令都是共享这个进程

jobs.jobid.steps.directory
    定义执行的目录	
jobs.jobid.steps.with
    aciton执行的参数。kv列表。在action里面可以定义参数。以INPUT_开头
jobs.jobid.steps.with.args
    传递给docker 的ENTRYPOINT 的参数。 参数在docker的CMD instruction中使用。
jobs.<job_id>.steps.with.entrypoint	 
jobs.<job_id>.steps.env  runner的环境
jobs.<job_id>.steps.continue-on-error 忽略错误继续执行，true。
jobs.<job_id>.steps.timeout-minutes 执行超时，kill的时间。

配置好之后。有ui界面进行管理。 查看日志，下砸日志，搜索日志。查看历史运行。等等：https://help.github.com/en/actions/configuring-and-managing-workflows/managing-a-workflow-run#viewing-your-workflow-history

工作流的了解事项： 触发条件 event-activity-limit 工作流组的定义与组合 jobs-job-step-run step的表达式语法和上下文变量：https://help.github.com/en/actions/reference/context-and-expression-syntax-for-github-actions runner机器的配置。尤其是文件系统的配置, 仓库位置。 runner实例处理step的。 run的用法，命令，acion脚本，容器 工作流的web使用。 查看，搜索，下载，日志。 管理执行流程。
